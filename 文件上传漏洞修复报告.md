### 1.3 原始漏洞代码

```python
# app.py 第 584-597 行（修复前）
if request.method == "POST":
    file = request.files.get("file")
    if not file or file.filename == "":
        return render_template("upload.html", error="请选择要上传的文件")

    # ❌ 漏洞核心：无任何安全校验
    filename = file.filename                              # 原始文件名，含 ../ 和特殊字符
    filepath = os.path.join(UPLOAD_DIR, filename)         # 路径穿越未防御
    file.save(filepath)                                   # 直接保存，无类型检查、无重名检测
    # 无审计日志、无用户关联
```

这段代码的问题在于：**攻击者完全控制 `filename` 变量**（来自 HTTP 请求的 `Content-Disposition` 头），而代码将其直接传递给 `os.path.join()` 和 `file.save()`，未做任何安全校验。

### 1.4 攻击链全景

攻击者可组合利用多个漏洞实现从代码执行到持久化驻留的完整攻击链：

```
上传 webshell（U-01）→ 远程命令执行 → 读取 /etc/shadow
覆盖主程序（U-03+U-04）→ 植入后门代码 → 长期驻留
上传 XSS 页面（U-02）→ 窃取管理员 Cookie → 接管管理后台
```

## 2. 漏洞详情与修复方案

### 2.1 U-01 + U-02：任意文件上传与存储型 XSS（🔴 严重）

**CVSS 3.1 评分：** 9.1（Critical）— AV:N/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:H

**问题分析：** 代码未检查文件扩展名，任何类型的文件均可上传并通过 `/static/uploads/` 直接访问。CSP 响应头仅作用于 Flask 模板渲染的页面，不作用于直接访问的静态文件（Werkzeug 开发服务器不附加 CSP 头）。

**危害演示：**
```bash
# 攻击者上传 PHP webshell
curl -X POST http://target/upload -F "file=@shell.php" -F "_csrf_token=..."
curl http://target/static/uploads/shell.php?cmd=id          # RCE 成功

# 攻击者上传恶意 HTML 页面
curl -X POST http://target/upload -F "file=@phishing.html" -F "_csrf_token=..."
# 将 URL 发给其他用户 → 浏览器执行恶意脚本 → Cookie 被窃取
```

**修复措施：** 添加扩展名白名单校验，仅允许图片类型。

```python
ALLOWED_EXTENSIONS = {"png", "jpg", "jpeg", "gif", "webp"}

def is_allowed_file(filename):
    """校验文件扩展名是否在白名单内"""
    if "." not in filename:
        return False
    ext = filename.rsplit(".", 1)[1].lower()
    return ext in ALLOWED_EXTENSIONS
```

### 2.2 U-03：路径穿越（🟠 高危）

**CVSS 3.1 评分：** 7.5（High）— AV:N/AC:L/PR:L/UI:N/S:U/C:N/I:H/A:H

**问题分析：** `os.path.join()` 不对 `../` 做任何防御，攻击者可通过构造的文件名将文件写入任意目录。

```python
# 攻击者设置 filename = "../../../app.py"
os.path.join(UPLOAD_DIR, "../../../app.py")
# → /opt/Class01/static/uploads/../../../app.py
# → 真实路径: /opt/Class01/app.py（覆盖主程序）
```

**修复措施：** 使用 `os.path.basename()` 剥离所有目录路径，并结合正则过滤危险字符。

```python
def safe_filename(filename):
    """清理文件名：去除路径 + 过滤危险字符"""
    safe = os.path.basename(filename)                    # 去除 ../ 目录
    safe = re.sub(r"[^a-zA-Z0-9._-]", "_", safe)         # 危险字符替换为 _
    if not safe or safe.startswith("."):
        safe = f"upload_{uuid.uuid4().hex[:8]}"
    return safe
```

| 攻击输入 | 修复前 | 修复后 |
|---------|--------|--------|
| `../../../etc/passwd.png` | 写入 `/opt/etc/passwd.png` | `basename()` → `passwd.png`，安全写入 uploads/ |
| `..%5c..%5cconfig.py` | 反斜杠路径穿越 | 正则将 `\` 替换为 `_`，`__config.py` |

### 2.3 U-04：无重名检测（🟠 高危）

**CVSS 3.1 评分：** 6.5（Medium）— AV:N/AC:L/PR:L/UI:N/S:U/C:N/I:H/A:N

**问题分析：** `file.save()` 直接覆盖已存在的同名文件。攻击者可利用此缺陷：
1. 替换已被系统管理员验证合法的头像文件
2. 多次上传同名文件导致合法用户数据丢失
3. 结合路径穿越覆盖系统关键文件后，再以正常上传掩盖痕迹

**修复措施：** 保存前检测文件是否存在，冲突时追加 UUID 随机后缀。

```python
if os.path.exists(dest_path):
    name_part = safe_name.rsplit(".", 1)[0]
    ext_part = safe_name.rsplit(".", 1)[1] if "." in safe_name else ""
    safe_name = f"{name_part}_{uuid.uuid4().hex[:8]}.{ext_part}" if ext_part \
                else f"{name_part}_{uuid.uuid4().hex[:8]}"
    dest_path = os.path.join(UPLOAD_DIR, safe_name)
```

### 2.4 U-05：文件名特殊字符

**问题分析：** 文件名中的 `'` `"` `;` `|` 等特殊字符若不经处理直接写入文件系统，可能导致日志解析异常、CSV 注入等问题。

**修复措施：** 统一使用 `re.sub(r"[^a-zA-Z0-9._-]", "_", filename)` 过滤，与 U-03 共用 `safe_filename()` 函数。

| 用户原始文件名 | 修复后磁盘文件名 |
|---------------|----------------|
| `test'.png` | `test_.png` |
| `a\|b.png` | `a_b.png` |
| `test' UNION SELECT --.png` | `test__UNION_SELECT___--.png` |
| `my cat.png` | `my_cat.png` |

> 数据库 `uploads` 表的 `original_name` 字段保留用户原始文件名，用于审计追溯。

---

### 2.5 U-06：文件不关联用户

**问题分析：** 上传的文件不与任何用户关联，无法追溯恶意文件的来源，也无法支持用户管理自己上传的文件。

**修复措施：** 新建 `uploads` 数据表记录每次上传操作。

```sql
CREATE TABLE IF NOT EXISTS uploads (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL,         -- 上传者
    filename TEXT NOT NULL,         -- 磁盘上的安全文件名
    original_name TEXT NOT NULL,    -- 用户提交的原始文件名（审计用）
    file_size INTEGER DEFAULT 0,
    upload_time TEXT NOT NULL
);
```

每条上传记录均包含完整的审计信息，与用户绑定。日志示例如下：

```
id=3  username=admin  filename=valid_9f46f993.png  original_name=valid.png
                                                      ↑ 同名文件被自动重命名
```

---

### 2.6 U-07：流式上传绕过大小限制

**问题分析：** Flask 的 `MAX_CONTENT_LENGTH` 不限制流式（chunked transfer）请求，攻击者可发送超大文件导致服务器内存耗尽。

**修复措施：** 在应用层额外手动校验 Content-Length，形成双重防御。

```python
# 第 1 层：Flask 框架层（限制常规请求）
app.config["MAX_CONTENT_LENGTH"] = 16 * 1024 * 1024

# 第 2 层：应用层（防御流式绕过）
content_length = request.content_length or 0
if content_length > 16 * 1024 * 1024:
    return render_template("upload.html", error="文件大小超过16MB限制")
```

### 4.5 安全使用建议

| 场景 | 建议 | 优先级 |
|------|------|:------:|
| **生产部署** | 在 Nginx 层对 `/static/uploads/` 配置 `add_header X-Content-Type-Options nosniff`，防止浏览器 MIME 嗅探 | 🔴 高 |
| **生产部署** | 配置 Nginx 禁止执行 uploads 目录下的脚本：`location /static/uploads/ { location ~ \.php$ { deny all; } }` | 🔴 高 |
| **扩展类型** | 如需支持 SVG 上传，必须同时配置 CSP 禁止 script 标签，防止 SVG 矢量图内的 XSS 攻击 | 🟠 中 |

| **文件清理** | 定期清理 `uploads` 表中超过 30 天的记录及对应磁盘文件，释放存储空间 | 🟡 低 |
| **磁盘限制** | 为 `static/uploads/` 目录配置磁盘配额（如 Linux `quota` 或容器 volume 大小限制） | 🟡 低 |
| **安全审计** | 定期检查 `uploads` 表的 `original_name` 字段，识别异常文件名模式 | 🟡 低 |
