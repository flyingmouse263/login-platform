| 编号 | 漏洞名称 | 类型 | 严重程度 | 代码位置 |
|:----:|---------|------|:--------:|---------|
| P-01 | 路径穿越（LFI） | 任意文件读取 | 🔴 严重 | `app.py:825` |
| P-02 | 未限制登录 | 权限缺失 | 🟠 高危 | `app.py:816-844` |

### 1.2 原始漏洞代码

```python
# 修复前（app.py）
@app.route("/page", methods=["GET"])
def page():
    name = request.args.get("name", "")
    if name:
        # ❌ P-01: 直接拼接，无路径校验
        page_path = os.path.join("pages", name)
        if os.path.exists(page_path):
            with open(page_path, "r") as f:
                page_content = f.read()      # ❌ P-02: 无登录校验
        else:
            page_path_html = page_path + ".html"
            if os.path.exists(page_path_html):
                with open(page_path_html, "r") as f:
                    page_content = f.read()
```

## 2. 漏洞详情与修复

### 2.1 P-01：路径穿越（LFI）（🔴 严重）

#### 漏洞成因

```python
page_path = os.path.join("pages", name)
```

这行代码存在三个层面的缺陷：

**① `os.path.join()` 不防御 `../`**

`os.path.join("pages", "../app.py")` 的结果是 `"pages/../app.py"`。`os.path.join` 的语义是"沿着路径走"，遇到 `..` 会正常拼接，不会截断或报错。这个路径在文件系统上等价于 `app.py`。

```
os.path.join("pages", "help")       → "pages/help"          ✅ 正常
os.path.join("pages", "../app.py")  → "pages/../app.py"     ❌ 等价于 app.py
os.path.join("pages", "../../../etc/passwd")
                                    → "pages/../../../etc/passwd"  ❌ 等价于 /etc/passwd
```

**② 用户输入完全可控**

`name = request.args.get("name", "")` 从 URL 查询参数直接获取，攻击者可传入任意字符串。结合第①点，`name` 中的 `../` 可以任意穿越目录层级。

**③ 读取后未做任何限制**

`open(page_path, "r")` 读取文件内容后直接渲染到页面，无内容过滤、无路径白名单。攻击者可以利用这个漏洞读取服务器上的任何文本文件。

---

#### 攻击路径推演

| 阶段 | 攻击步骤 | 结果 |
|:----:|---------|------|
| 1 | `GET /page?name=../app.py` | 获取应用源码，找到 `SECRET_KEY`、数据库路径、其他路由逻辑 |
| 2 | `GET /page?name=../../../etc/passwd` | 获取系统用户列表，确认存在哪些可登录账号 |
| 3 | `GET /page?name=../../../etc/shadow` | 获取密码哈希，离线破解后可 SSH 登录服务器 |
| 4 | `GET /page?name=../data/users.db` | 获取 SQLite 数据库，提取所有用户密码和手机号 |
| 5 | `GET /page?name=../.env` | 获取环境变量配置（若存在 .env 文件） |

从第 1 步到第 4 步，攻击者可以在 **1 分钟内** 完成从信息收集到凭据窃取的全过程。

---

#### 攻击验证（实际测试结果）

```bash
# ① 读取应用源码
GET /page?name=../app.py
→ 响应中包含 app.secret_key、SECRET_KEY、validate_env 等关键源码

# ② 读取系统用户列表
GET /page?name=../../../etc/passwd
→ 响应中包含 root:x:0:0:root:/root:/bin/bash 等系统用户条目

# ③ 读取密码哈希
GET /page?name=../../../etc/shadow
→ 响应中包含 root:$y$j9T... 等加密密码
```

---

#### 修复方案

核心思路：**将用户输入映射到受控目录内，再校验映射后的路径是否越界。**

```python
# 第 1 步：确定允许访问的根目录（pages/ 的绝对路径）
safe_base = os.path.realpath(os.path.join(app.root_path, "pages"))

# 第 2 步：将用户输入映射到 pages/ 目录下，解析为绝对路径
requested_path = os.path.realpath(os.path.join(safe_base, name))

# 第 3 步：校验解析后的路径是否以 pages/ 开头
if not requested_path.startswith(safe_base + os.sep):
    log_security_event("PAGE_TRAVERSAL", username, request.remote_addr,
                       f"路径穿越尝试: {name}")
    page_error = "页面不存在"
```

**为什么这种方案有效？**

`os.path.realpath()` 会解析路径中的所有符号链接和 `..`，输出一个**不含任何相对路径成分的绝对路径**。然后通过 `startswith` 检查这个绝对路径是否以 `safe_base`（`pages/` 的绝对路径）开头，只要 `..` 穿越到了 `pages/` 以外的目录，检查就会失败。

---

#### 修复原理图解

```
用户输入 name = ../app.py

                                os.path.join()              os.path.realpath()
safe_base = "pages/"  ──────────┬────────────────────────────────────────────
                                │
name = "../app.py" ─────────────┤
                                ↓
                "pages/../app.py"  ──────────→  "/opt/Class01/app.py"
                                                      ↓
                                          startswith("/opt/Class01/pages/") ?
                                                      ↓
                                                    ❌ 拒绝

──────────────────────────────────────────────────────────────────────────────

用户输入 name = help

                                os.path.join()              os.path.realpath()
safe_base = "pages/"  ──────────┬────────────────────────────────────────────
                                │
name = "help" ──────────────────┤
                                ↓
                     "pages/help"  ──────────→  "/opt/Class01/pages/help.html"
                                                      ↓
                                          startswith("/opt/Class01/pages/") ?
                                                      ↓
                                                    ✅ 通过
```

### 2.2 P-02：未限制登录（🟠 高危）

**问题分析：** `/page` 路由未检查 session 登录状态，攻击者无需任何身份认证即可读取服务器文件。

**攻击验证：** 无 Cookie 直接访问 `/page?name=help` → 页面正常渲染。

**修复方案：** 路由入口添加 session 登录校验。

```python
username = session.get("username")
if not username:
    return redirect("/login")
```

**验证结果：** ✅ 未登录用户访问 `/page` 时 302 跳转到 `/login`。

### 4.2 修复后的处理流程

```
请求 /page?name=...
  → 登录校验（P-02）
      → 未登录？→ 302 跳转 /login
      → 已登录 → realpath 路径校验（P-01）
          → 不在 pages/ 内？→ "页面不存在"

              → 存在？→ 渲染内容

```
